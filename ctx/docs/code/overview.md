# Инженерные нормы уровня code

Path: `./ctx/docs/code/overview.md`

Каталог `ctx/docs/code` фиксирует *инженерные* правила работы с исходниками без повторения смыслов продуктового, архитектурного или constraints-уровней. Он ссылается на них (например, `ctx/docs/architecture/overview.md`) за описанием контуров `client`, `signal`, `server`, `direct-interaction`, но сам использует контуры только как организационные категории, группируя код и тесты под общим именем.

Контурные каталоги в `ctx/docs/code` описывают способы построения модулей, DI-зависимости, namespace и runtime-поведение конкретного слоя реализации. Общие нормы кладутся в `shared/`. Перечень точек входа уровня code:

- `shared/` — универсальные соглашения по JavaScript, DI, namespace и тестированию, применимые ко всем контурам.
- `client/` — код клиентского PWA: структура `web/app`, DI-контракты, правила модуля `Flow`, UI и хранилища.
- `signal/` — N/W-инфраструктура сигналинга: серверные и клиентские модули WebSocket, управление сессиями и сообщение.
- `server/` — Node.js-композиция: `bin/server.js`, `src/Back`, DI-контракты, шаблоны модулей, логирование.
- `direct-interaction/` — взаимодействие с медиа и WebRTC (Media, Peer, Session) и связанные модули.

Документы уровня code должны:

1. Описывать **только** кодовые практики (DI, namespace, логика модулей, тесты, логирование, PWA-инфраструктура), не вводить новые архитектурные сущности, не пересказывать state machine или поведении по умолчанию.
2. Ссылаться на архитектурные/продуктовые/constraints-документы, а не пытаться переопределить их смысл (например, `ctx/docs/architecture/overview.md` для контуров, `ctx/docs/product/overview.md` для пользовательского языка, `ctx/docs/constraints/storage.md` для ограничений хранения).
3. Группировать документацию по контурам: shared-правила — для всех, контурные папки — только для соответствующего слоя.
4. Быть декларативными, компактными и технологически нейтральными в духе ADSM.

Иерархия документации не допускает дублирование архитектурных терминов: если смысл приходится уточнять, делайте ссылку вроде «см. `ctx/docs/architecture/...`» и оставляйте здесь только code-уровневые детали.
