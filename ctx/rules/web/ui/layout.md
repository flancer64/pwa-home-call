# Приложение: компоновка и крупный ритм интерфейса

## Назначение

Документ закрепляет единый каркас пользовательского пространства Приложения: минималистичный трёхзонный ритм для `home`, `invite`, `call` и `end`, который подчёркивает одну задачу на экран, подчёркивает крупные контролы и не допускает «шума» вокруг действий.

## Трёхзонный скелет

- **Header** — бренд, краткий статус или ссылка на сервисную карточку. Текст централизованно информирует, но не превращается в дополнительную инструкцию; любые пояснения оформлены как лаконичные строки, привязанные к конкретному экрану.
- **Action** — единственный крупный CTA и опционально вспомогательное действие (закрытие карточки, повторный запуск). Элементы используют `component/big-button`, не содержат полей ввода и не конкурируют за внимание; все второстепенные элементы маскируются как иконки в углу.
- **Hint** — зона для короткого контекста (одна-две строки). Она появляется только при необходимости и содержит только факт («Ссылка готова», «Собеседник подключён») без пошаговых инструкций. Не превращается в описание состояний или повторяется на других экранах.

Трёхзонная структура реализуется через `component/screen-card`, который формирует фон, границы и три слота для `component/screen-header`, `component/big-button` и `component/screen-note`. Каждый экран описывает только уникальное содержание зон, подставляя компоненты и связывая их с текстом; общие ограничения и визуальный ритм импортируются из `style.md`.

## Исключение: `call`

`call` выводит собственную компоновку: основной слой — `remoteStream`, который занимает всю площадь, а `localStream` располагается в плавающей миниатюре правого верхнего угла. Внизу — две круглые FAB-кнопки (`красная крест` и `зелёная шестерёнка`); больше никаких зон `header/action/hint`. Только `call` использует этот layout, все остальные экраны полагаются на `component/screen-card`. FAB доступен исключительно в `call`; другие экраны не имеют плавающих настроек.

## Статусы и переходы

Статусы медиа, разрешений и сигналинга никогда не рисуются как слои UI — они транслируются исключительно в тосты (`notifications.md`). Экран `call` остаётся плоским, переходы инициируются событиями `onEndRequest`/`onSettingsRequest`, а детали исполнения прописаны в `screens.md` и `core.md`.

Переходная логика (`home → invite → call → end` и сервисные карточки) фиксируется в `screens.md`; отдельные экраны упоминают только своё действие и связанные уникальные элементы.
