# Каталог `ctx/product/features/`

## Назначение

Каталог `features/` фиксирует продуктовые возможности Колобок на уровне «что делает приложение», не описывая их по шагам сценария и не погружаясь в техническую реализацию.

Он связывает сводный список в `../features.md` с более подробными артефактами (экраны, состояния, требования к пользовательскому опыту (UX)), которые затем используются на архитектурных уровнях.

---

## Структура каталога

- `AGENTS.md` — текущий файл, объясняющий назначение каталога и принципы работы с ним.
- `invite.md` — спецификация возможности приглашения по ссылке: путь «главный экран → приглашение → звонок», участники, поток и ожидаемая польза для пожилых пользователей.

При добавлении новых возможностей:

- для каждой отдельной возможности создаётся файл `<feature>.md`;
- документ описывает участников, поток, ключевые состояния интерфейса и связь с пользовательскими сценариями;
- подробные пошаговые последовательности выносятся в `../scenarios/`, чтобы избежать дублирования.

---

## Рекомендации для агента

- Если нужно описать **отдельную функцию или экран** (например, поведение экрана приглашения, индикаторов, кнопок) — добавляй или уточняй документы в каталоге `features/`.
- Если нужно описать **путь пользователя через несколько экранов** — используй каталог `../scenarios/` и ссылайся на соответствующие документы `features/*.md`.
- Избегай дублирования пошаговых сценариев в `features/`: здесь фиксируются ключевые состояния и требования, а не полный сценарий использования.

## Терминология

- При описании возможностей опирайся на `../terms.md` в каталоге `product/` — именно оттуда берутся понятия сеанса, ссылки приглашения, инициатора и получателя.
- Все термины интерфейсных элементов должны соответствовать словарю, чтобы документация `features/` была согласованной с остальным продуктовым уровнем.
